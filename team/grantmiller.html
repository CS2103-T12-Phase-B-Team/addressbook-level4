Autocomplete

Pressing &tab; will attempt to autocomplete current command. Turns text red if not suitable command exists. Since v1.2

To see parameters for a command, press tab on a complete command. Pressing tab again will remove the parameters. If the command has no parameters then the text will turn red. Since v1.3

If there are multiple possible commands for a given input, i.e. 'e' could be 'edit', 'exit', or 'export', then a dropdown box will appear with the possible commands. These can be navigated using mouse or arrow keys and enter to select the desired command. Since v1.5

End of Extract

Justification

Autocomplete is a common feature among many applications. It greatly increases the ease of use for a user by speeding up their ability to complete common commands.

My implementation use a dropdown box to visually show the user what possible commands there are for an input, if any, as well as easily enabling the user to see the parameters for their command directly in the command line.

Start of Extract [from: Developer Guide]

Autocomplete

The autocomplete functionality is accomplished by using a character trie to store all possible commands. The trie has a very simple set of commands, shown below.

public interface Trie {
    String attemptAutoComplete (String input);
    void insert (String input);
    Set<String> getCommandSet();
}

The trie is made up of node classes which only have three fields. The next field represents a character on the same level as the current one. The child represents a next possible letter. For example in the words edit and exit the node for 'd' would have a child 'i' and a next 'x'. 'x' would have a child 'i'.

public class Node {

    private Node next;
    private Node child;
    private char key;
    ...
}

Upon the pressing of the tab key, the command box calls attemptAutoComplete, in the commandTrie class. This method will traverse the trie using the content of the command box. If the input has a character not present in the trie, then the attempt fails. If the trie parses through the input but reaches a node with a next option, it will attempt to provide the user with possible commands for a given input. In this case, a dropdown box will appear in the UI that lists the possible commands for the user. This is accomplished using a recursive searching of the trie that follows each possible path and returns a set of possible commands.

 private List<String> recursiveGetOptions(Node start, String stub) {
        List<String> options = new ArrayList<>();
        StringBuilder output = new StringBuilder();
        output.append(stub);

        while (!isLeaf(start)) {
            if (start.hasNext()) {
                options.addAll(recursiveGetOptions(start.getNext(), output.toString()));
                if (start.hasChild()) {
                    output.append(start.getKey());
                    options.addAll(recursiveGetOptions(start.getChild(), output.toString()));
                }
                return options;
            } else {
                output.append(start.getKey());
                start = start.getChild();
            }
        }
        output.append(start.getKey());
        options.add(output.toString());
        return options;
}

If the trie parses the input and reaches a node with only children, it will autocomplete the input to a valid command. If the autocomplete fails, the text is turned red.

An added functionality is that if the user presses tab again, the command will add a usage framework for the command, i.e. find will be autocompleted to find [names]. This helps the user to understand command usage. If tab is pressed again, the input simply reverts back to the command.

One class, command collection, is used to hold the set of commands and a map from each command word to the command usage. This is used by the trie to both construct the command trie and autocomplete from a command word to a command word and its usage. This static class simply contains a set of each command’s command word that is statically initialized and used by the commandTrie class.

Justification

Groups were added to the addressbook so that the owner could easily filter their addressbook between groups of people.

Implementation

Start of Extract [from: Developer Guide]

Groups

The groups feature is implemented in every component of the addressbook. In the logic section, there are two new commands: group and filter. The group command takes care of creating, adding to, and deleting a group. The default group selection is "none", and it cannot be deleted. If a group is deleted, then a person is assigned the none group.

In the model section, there is a new field in the person object for a group. This object simply contains the group name. In the addressbook, class there’s now a uniqueGroupList. This field is similar to the implementation for the Person List or Tag list. It keeps track of all groups within the addressbook and is used to sync when new groups are added or groups are deleted.

In storage, the xml files have been updated to include the uniqueGroupList elements (XMLAdaptedGroup), as well as storing the group field for each person.

In UI, there’s now a list view that in a split panel that shows the menu on the left, and each created group on the right. Selecting a group will update the filtered list view to display only people in the selected group.

Design Considerations

Aspect: Implementation of Groups vs Using Tags
Alternative 1 (current choice): Use Groups to filter view
Pros: Groups and tags can provide a more dynamic way of sorting your addressbook
Cons: Need whole new data structures to store groups and UI interfaces for presenting groups to user
Alternative 2: Use tags to filter view
Pros: Easy to implement in UI, easy to implement filter
Cons: Tags are used to describe a person, while groups should be used to categorize. For example a tag may be friend, while the group would be Americans or Singaporeans. Having groups separate from tags allows for a more nuanced use of addressbook.

Aspect: Implementation of Groups using UniqueGroupList
Alternative 1 (current choice): Use UniqueGroupList and add method for existence in list+ Pros: Fits the current model for Persons and Tags and adds an easy way to check if a group exists without raising an exception+ Cons: Deviates a bit from the past implementation for the other uniqueLists+ Alternative 2: Use UniqueGroupList similar to UniquePersonList+ Pros: Fits model for other structures in addressbook+ Cons: Has some exceptions that make code cluttered and unclear+ Alternative 3: Use basic List Structure+ Pros: Easy to implement+ Cons: Would require extensive modifications to fit the existing model for addressbook

Aspect: Placement of group selection listener
Alternative 1 (current choice): Placed in MainWindow and modified CommandBox for CLI-less command execution
Pros: Allows transparent execution of filter command and output into the result box using existing model as well as keeps listeners in the same class
Cons: Overloads execution method and has a public method in CommandBox for executing command
Alternative 2: Place listener in CommandBox
Pros: Keeps the listener in the same class as the execute method
Cons: Clutters CommandBox by adding a listener to a class that doesn’t have any
